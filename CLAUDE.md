# CLAUDE.md

Guidance for AI assistants (Claude Code, Amp, etc.) working in this repo.

## Project overview

Personal portfolio site at **amirsalarsafaei.com**. Next.js frontend +
Rust (Tonic) backend, communicating via gRPC-Web, backed by PostgreSQL.

**Monorepo layout:**

```
backend/        Rust + Tonic service (gRPC + image upload + sqlx)
frontend/       Next.js 14 app
proto/          .proto definitions (managed by buf)
autogenerated/  Generated gRPC stubs — TS package + Rust crate (checked in)
.sqlx/          sqlx offline query cache (workspace root)
```

**Infrastructure:** PostgreSQL, uploads on a Docker volume, deployed via
Docker Compose. CI publishes images to GHCR.

## Tooling

- **Dev environment** — a single [devbox](https://www.jetify.com/devbox)
  config at the repo root (`devbox.json`) provisions Rust (via rustup), Node
  20, Yarn, buf, sqlx-cli, protobuf, postgres client, openssl, treefmt,
  prettier. Devbox is the only supported dev/deploy toolchain.
- **`nix/` + `flake.nix` (legacy, do not extend)** — these exist *only* so an
  external dotfiles repo can consume this repo as a flake input and build the
  frontend/backend from source. This is a **known bad pattern**: it couples
  the two repos through a push + `flake.lock` bump + rebuild round trip and
  reintroduced a flake into a repo that is otherwise Docker-only. Do not build
  new tooling on it; the intended direction is to deploy the published GHCR
  image instead and drop the flake input entirely.
- **Direnv** — `.envrc` runs `use devbox` and loads `.env` if present.
- **Task runner** — [`just`](https://github.com/casey/just); recipes live in
  `justfile`. Do not reintroduce a Makefile.
- **Formatting** — `treefmt.toml` drives rustfmt + prettier in one pass
  (`just format` or `treefmt`).

## Commands (run inside `devbox shell`)

Tasks are defined in **`justfile`**. Run `just` with no args to list them.

```bash
just                   # list every recipe
just buf-generate      # regenerate gRPC code after editing proto/
just build             # cargo build && yarn build
just test              # cargo test --workspace
just lint              # clippy + next lint
just format            # treefmt

just docker-build      # build all images
just up                # docker compose up -d
just dev               # postgres + backend + frontend-dev (hot reload)
just down

just db-migrate        # sqlx migrate run (needs $DATABASE_URL)
just db-prepare        # refresh .sqlx/ after editing query macros
```

## Architecture

### Backend (Rust / Tonic) — `backend/`

- `src/main.rs` — entry point, wires gRPC services + HTTP routes.
- `src/blogs.rs`, `src/tags.rs`, `src/spotify.rs`, `src/imageserver.rs` —
  gRPC/HTTP handlers.
- `src/db/` — sqlx query layer (compile-time checked via `.sqlx/`).
- `src/config.rs` — loads TOML config (`config.toml`, `spotify.toml`) and
  overlays env vars; see `backend/config.example.toml`.
- `src/toproto.rs` — domain → protobuf conversions.
- Migrations in `backend/migrations/` (sqlx-format, applied by the
  `migrate` compose service on startup).

### Frontend (Next.js) — `frontend/`

- `src/app/` — App Router pages (`blogs/`, `admin/`, `playground/`,
  `aboutme/`, `resume/`).
- `src/clients/grpc.ts` + `src/providers/GrpcProvider.tsx` — gRPC-Web setup.
- Imports generated stubs from `autogenerated/ts/`.
- Standalone Next.js build output is what the Dockerfile copies.

### Proto / gRPC

- `proto/blogs/blogs.proto`, `proto/playground/spotify.proto`.
- After editing, run `make buf-generate` to regenerate both `autogenerated/ts/`
  and `autogenerated/rust/src/`.

### Deployment

- `compose.yaml` — production-style stack (postgres + migrate + backend +
  frontend).
- `compose.override.yaml` — adds `frontend-dev` under the `dev` profile.
- `backend/Dockerfile`, `frontend/Dockerfile`, `frontend/Dockerfile.dev`.
- `.github/workflows/build-publish.yml` — multi-arch image publish to GHCR.

## Important patterns

1. **gRPC is the only API** — frontend never calls REST; everything goes
   through the generated TS stubs in `autogenerated/ts/`.
2. **sqlx offline mode** — compile-time query checking uses `.sqlx/` at the
   workspace root. Regenerate with `make db-prepare`.
3. **Config priority** — env vars (from `.env` / compose) override values in
   `config.toml`. In containers prefer env vars.
4. **Admin auth** — token-based; the backend's `AUTH_TOKEN` must match what
   the admin UI sends.

## Don'ts

- Don't extend the legacy `flake.nix` / `nix/` (see Tooling) — Docker/GHCR is
  the intended deployment target; the flake is a bad pattern to be removed,
  not built upon.
- Don't commit `.env`, `backend/config.toml`, `backend/spotify.toml`,
  `backend/uploads/`.
- Don't hand-edit anything under `autogenerated/` — always go through buf.
