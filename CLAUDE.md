## Project Overview

Personal portfolio website at amirsalarsafaei.com with a **Next.js frontend** and **Rust (Tonic) backend**, communicating via **gRPC**. The site features a 3D laptop environment, blog CMS, music player, and admin panel.

**Monorepo layout:**
- `frontend/` - Next.js app (React, TypeScript, Three.js, gRPC-Web)
- `backend/` - Rust server (Tonic, PostgreSQL via sqlx, cornucopia query builder)
- `proto/` - `.proto` definitions for blogs and spotify services
- `autogenerated/` - gRPC code generated by buf (ts + prost/tonic)
- `nix/` - Nix flake for dev shell, builds, and container images

**Infrastructure:** PostgreSQL database, images stored via a dedicated image server, deployed via Podman container images built with Nix.

## Development Commands

```bash
# Enter dev shell (all dependencies: rust, node20, yarn, protobuf, buf, postgres, podman)
nix develop

# Generate gRPC code (run after any .proto changes)
make buf-generate

# Build backend binary
nix build .#backend

# Build frontend
nix build .#frontend

# Build Docker images
nix build .#backendImage
nix build .#frontendImage

# Lint/format (via treefmt in nix shell)
treefmt --fail-on-change .

# Deploy to VPS
make deploy
```

## Architecture

### Backend (Rust/Tonic)

`backend/src/main.rs` - Entry point: registers gRPC services, HTTP routes, and starts the server.

Key modules:
- `blogs.rs` - Blog CRUD handlers (via `db/blogs.rs` with cornucopia queries)
- `tags.rs` - Tag management
- `spotify.rs` - Spotify integration
- `imageserver.rs` - Image upload/serving
- `config.rs` - TOML config loading (auth token, DB, server, image server)
- `db/` - sqlx-based database layer with cornucopia query crates
- `toproto.rs` - Domain-to-protobuf conversion helpers

Database migrations live in `backend/migrations/`. Uses sqlx for compile-time query checking.

### Frontend (Next.js)

`frontend/src/app/` - Next.js App Router structure:
- `blogs/` - Blog listing + individual blog pages with rich content rendering
- `admin/` - Admin panel with blog editor (Auth-protected)
- `playground/` - Interactive 3D/animation demos
- `aboutme/`, `resume/` - Static pages

Key infrastructure:
- `frontend/src/clients/grpc.ts` - gRPC-Web client setup
- `frontend/src/providers/GrpcProvider.tsx` - React context wrapping gRPC client
- `frontend/src/components/BlogsList/` - Blog list component

The frontend uses gRPC-Web to call the backend's Tonic services. The `autogenerated/ts/` directory contains the TypeScript gRPC stubs.

### Proto / gRPC

Two proto services:
- `proto/blogs/blogs.proto` - Blog CRUD RPCs
- `proto/playground/spotify.proto` - Spotify streaming RPCs

After editing `.proto` files, run `make buf-generate` to regenerate both TypeScript and Rust code.

### Nix

`flake.nix` uses flake-parts with:
- `devShells.default` - Dev environment (rust, node20, yarn, buf, postgres, podman)
- `packages.backend`, `packages.frontend` - Binaries
- `packages.backendImage`, `packages.frontendImage` - OCI images
- `formatter` - treefmt (prettier, rustfmt, etc.)
- `checks.formatting` - CI formatting check

## Key Patterns

1. **gRPC as primary API** - Frontend never calls REST; all data goes through gRPC-Web stubs in `autogenerated/ts/`.
2. **Cornucopia for DB queries** - SQL files are compiled into type-safe query functions. Check `backend/src/db/` for the generated code.
3. **TOML config** - Backend reads from `/config` mount in production (see `DEPLOYMENT.md` for config format).
4. **Admin auth** - Protected by token-based auth; the `auth_token` from backend config is used.
